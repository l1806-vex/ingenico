1.           Introduction Up

IngEstate Parameter Management is a new module introduced in IngEstate v7.2. Parameter Management concerns the definition, configuration, storage, and downloading of parameters associated with a specific terminal application.

 

An important source input into the Parameter Management functionality is the parameter template file. This is an XML file responsible for detailing the following:

 

                     General information about the template.

                     Parameter definitions & their validation rules.

                     Parameter layout rules for MMI clients.

                     File generation rules (where ultimately the files will be sent to a terminal).

 

Although these individual sections will generally be produced by different people at different times, they are currently packaged together in the same XML file for convenience.

1.1.       Purpose

The purpose of this document is to clearly define the expected format of a parameter template XML file, in order that they may be created and actively maintained by the relevant people. This document will not describe any other part of the Parameter Management process.

1.2.      Scope

This document will be of use to all people involved in the production of a Parameter Template; namely terminal application developers and IngEstate administrators.

1.3.      References

None.

1.4.     Glossary

 

API Application Programming Interface

DB Database

MMI Man Machine Interface (Graphical Client)

XML Extensible Markup Language

 

2.         Parameter Template Definition Up

A parameter template XML file is used to define parameters for one or more versions of a terminal application. If a terminal application is updated without the need for new parameters, then the parameter template itself will not need to change.

 

Note that currently a parameter template file has no direct reference to the specific terminal application and versions it is concerned with (at best, there may be an indirect reference in the template description), and it is the responsibility of the IngEstate administrators to ensure that the correct templates are sent to the correct terminals.

 

The basic format of a parameter template file is as follows:

 

<template>

 

<fields>

<!--

Field Definition Section (see Section 3)

Defines all parameters (& their validation rules) required for the template

-->

</fields>

 

<layout>

<!--

Layout Definition Section (see Section 4)

Defines how parameters should be displayed in a MMI

-->
</layout>

 

<output>

<!--

Output Definition Section (see Section 5)

Defines how parameters should be written to files for sending to a terminal -->

</output>

 

</template>

 

A small working example of a parameter template XML file can be found in Section 6.

2.1.      <template> Element

This is the main root element for the parameter template xml file.

2.1.1.   Attributes

The <template> element has the following attributes:

 

Attribute Name

Description

Required

id

String detailing the id of the template.

 

The template id and version uniquely identify the template in IngEstate.

Yes

version

String detailing the version of the template. This can be up to five characters in length.

 

The template id and version uniquely identify the template in IngEstate.

Yes

name

String detailing a user friendly name for the template. This may be used by a MMI for display to a user.

Yes

description

String detailing a description for the template. This may be used by a MMI for display to a user.

No

xmlns

Fixed string detailing the default namespace of the XML elements and attributes.

 

The Value must be "http://www.ingenico.com/ingestate/paramTemplate"

 

Yes

2.1.2.  Nested Elements

The <template> element has exactly one of each of the following nested elements, in the order specified.

 

Element Name

Description

Required

fields

This element details the parameters applicable to this template, along with their basic validation rules.

Yes

layout

This element details how the parameters should be presented in a MMI.

Yes

output

This element details how the parameters should be written to files for sending to a terminal.

Yes

 

 

3.         Field Definition Section Up

The field definition section details all parameters applicable to this template, along with their basic validation rules.

 

Note: A parameter validation check is always performed for stored parameter values immediately prior to parameter files being generated for sending to a terminal. If any parameter fails a validation check, then the whole parameter set is deemed invalid and no files will be sent.

 

The field definition section is usually created by the terminal application developer.

3.1.      <fields> Element

This is the main element for the field definition section.

3.1.1.   Attributes

The <fields> element has no attributes.

3.1.2.  Nested Elements

The <fields> element can contain any amount of the following nested elements, and in any order (not important to the functionality). As a minimum, at least one element must be supplied.

Element Name

Description

Required

dataField

This element details a standard data parameter.

No

choiceField

This element details a choice parameter. A choice parameter may have a value which can be chosen from a finite defined set.

No

structureField

This element details a structure parameter. A structure parameter consists of multiple sub-parameters (which may be individually configured), and may contain 1 or more rows. It is analogous to a table.

No

3.2.     <dataField> Element

The <dataField> element is used to define a standard single-value parameter. In a MMI, these types of parameters tend to be displayed in a free-form text entry box.

3.2.1.  Attributes

The <dataField> element has the following attributes:

 

Attribute Name

Description

Required

id

String detailing the unique id of the parameter. No two parameters in a template may share the same id.

Yes

name

String detailing a user-friendly name for the parameter. This may be used by a MMI for display to a user.

Yes

description

String detailing a user-friendly description for the parameter. This may be used by a MMI for display to a user.

No

access

String detailing the parameter access. Valid access options are as follows:

HIDDEN - Not to be shown to users of a MMI.
READONLY - Can be read by users of a MMI, but not edited.
READWRITE - Can be both read and edited by users of a MMI.

Yes

defaultValue

String representing the default value for the parameter in cases where a value has not been explicitly set and stored in the DB.

No

mandatory

Boolean that determines whether or not a value for the parameter is required to exist (either explicitly set and stored in the DB, or defined via the defaultValue attribute).

 

This attribute is used in the parameter validation check.

No

type

String detailing the type of the parameter. Valid options are as follows:

alphanumeric - Parameter can consist of any printable characters. Empty strings are allowed. (e.g. "abc123.-*")

letters - Parameter can only consist of alphabetical characters. Empty strings are allowed. (e.g. "abc")
numeric - Parameter can only consist of numerical characters. Empty strings are not allowed. (e.g. "123")
alphanumStrict - Parameter can only consist of alphabetical and numerical characters. Empty strings are allowed. (e.g. "abc123")
hexadecimal - Parameter can only consist of hexadecimal characters. Empty strings are not allowed. (e.g. "aaff0011")

 

If this attribute is not specified, then the parameter can be of any type.

 

This attribute is used in the parameter validation check.

No

minLength

Non-negative integer that specifies the minimum length (in characters) for the parameter.

 

This attribute is used in the parameter validation check.

No

maxLength

Positive integer that specifies the maximum length (in characters) for the parameter.

 

This attribute is used in the parameter validation check.

No

minValue

Integer that specifies the minimum value for the parameter. It is expected that the parameter is a valid number, and any non-numerical parameter will fail this check.

 

This attribute is used in the parameter validation check.

No

maxValue

Integer that specifies the maximum value for the parameter. It is expected that the parameter is a valid number, and any non-numerical parameter will fail this check.

 

This attribute is used in the parameter validation check.

No

regularExp

String detailing a regular expression pattern that the parameter must match.

For more information regarding regular expression construction, refer to the Pattern class in the Java 1.6 API (http://download.oracle.com/javase/6/docs/api/).

 

e.g.

"[01]{8}" will match a parameter value with exactly eight "0" or "1" characters - in any order.

 

"[^abc]" will match a parameter value with any character except "a", "b" or "c".

 

This attribute is used in the parameter validation check.

No

dateTimeFormat

String detailing a format for a date-time value that the parameter must match. Valid formats are as defined for the SimpleDateFormat class in the Java 1.6 API (http://download.oracle.com/javase/6/docs/api/).

 

As symbols are defined in the local of the IngEstate server JVM, it is recommended that only the numeric codes are used for the required sub-components of the date-time.

 

e.g.

"dd/MM/yyyy" will match a parameter value of "31/12/2012".

 

"YYYYMM" will match a parameter value of "201212".

 

"dd/MM HH:mm:ss" will match a parameter value of "31/12 22:00:00"

 

 

This attribute is used in the parameter validation check.

No

3.2.2. Nested Elements

The <dataField> element has no nested elements.

3.3.     <choiceField> Element

The <choiceField> element is used to define a single-value choice parameter. This is a parameter which may take on a value from a finite defined set. In a MMI, these types of parameters tend to be displayed as a drop-down box, or as a set of radio-buttons.

 

Note: The parameter validation check ensures that a choice parameter has a value matching one of its defined set.

3.3.1.  Attributes

The <choiceField> element has the following attributes:

 

Attribute Name

Description

Required

id

String detailing the unique id of the parameter. No two parameters in a template may share the same id.

Yes

name

String detailing a user-friendly name for the parameter. This may be used by a MMI for display to a user.

Yes

description

String detailing a user-friendly description for the parameter. This may be used by a MMI for display to a user.

No

access

String detailing the parameter access. Valid access options are as follows:

HIDDEN - Not to be shown to users.
READONLY - Can be read by users, but not edited.
READWRITE - Can be both read and edited by users.

Yes

defaultValue

String representing the default value for the parameter to assume in cases where a value has not been explicitly set and stored in the DB. Naturally, it only makes sense that this value matches one of the defined choices.

No

mandatory

Boolean that determines whether or not a value for the parameter is required to exist (either explicitly set and stored in the DB, or defined via the defaultValue attribute).

 

This attribute is used in the parameter validation check.

No

type

String detailing the choice type of the parameter. Valid options are as follows:

SINGLE - The parameter can take on the value of only one of the defined choices.

Yes

3.3.2. Nested Elements

The <choiceField> element can contain one or more of the following nested elements.

Element Name

Description

Required

choiceValue

This element details one valid choice for the choice parameter.

Yes

3.4.    <choiceValue> Element

The <choiceField> element is used to define a single choice for the choice parameter.

3.4.1.  Attributes

The <choiceValue> element has the following attributes:

 

Attribute Name

Description

Required

label

String detailing a user-friendly name for the choice. This may be used by a MMI for display to a user.

Yes

value

String detailing the actual value for the choice. This will be the value saved to the database, and if using standard IngEstate download commands, will be the actual value that ends up being sent down to the terminal.

Yes

3.4.2. Nested Elements

The <choiceValue> element has no nested elements.

3.5.     <structureField> Element

The <structureField> element is used to define a structure parameter. A structure parameter can be considered as analogous to a table. It will have one or more defined sub-parameters (columns), and can grow dynamically (consisting of zero or more rows).

3.5.1.  Attributes

The <structureField> element has the following attributes:

 

Attribute Name

Description

Required

id

String detailing the unique id of the parameter. No two parameters in a template may share the same id.

Yes

name

String detailing a user-friendly name for the parameter. This may be used by a MMI for display to a user.

Yes

description

String detailing a user-friendly description for the parameter. This may be used by a MMI for display to a user.

No

access

String detailing the parameter access. Valid access options are as follows:

HIDDEN - Not to be shown to users.
READONLY - Can be read by users, but not edited.
READWRITE - Can be both read and edited by users.

Yes

minRows

Positive integer that specifies the minimum number of rows that require to be populated for the structure.

 

The default value is 1.

 

This attribute is used in the parameter validation check.

No

maxRows

Positive integer that specifies the maximum number of rows that require to be populated for the structure.

 

The default value is 1.

 

This attribute is used in the parameter validation check.

No

 

3.5.2. Nested Elements

The <structureField> element can contain any amount of the following nested elements, and in any order. As a minimum, at least one element must be supplied.

 

The order of these elements is very important, as it determines in what order a structure's sub-parameters are written to generated files (as specified in the template's output definition section - see Section 5).

Element Name

Description

Required

dataField

This element details a standard data parameter as a structure sub-parameter.

 

This is as defined in Section 3.2, except that:

 

1. The access attribute cannot be set to HIDDEN (it will be treated as read-only).

2. The id of the sub-parameter defined by this element only needs to be unique only to the structure parameter (that is, two different structures may contain sub-parameters that have the same defined id).

No

choiceField

This element details a choice parameter as a structure sub-field.

 

This is as defined in Section 3.3, except that:

 

1. The access attribute cannot be set to HIDDEN (it will be treated as read-only).

2. The id of the sub-parameter defined by this element only needs to be unique only to the structure parameter (that is, two different structures may contain sub-parameters that have the same defined id).

No

 

4.         Layout Definition Section Up

The layout definition section details how the template parameters are to be displayed by the MMI (client interface) to the end user.

 

The following rules are applicable to the layout section:

 

                     Parameters can be displayed in any order, and do not need to follow the order as defined in the field definition section.

                     A parameter can only be displayed once in the layout.

                     A layout does not need to show all parameters defined in the field definition section (but a defined structure layout must show all of that structure's sub-parameters).

 

The layout definition section is usually created by the IngEstate Administrator.

4.1.     <layout> Element

This is the main element for the layout definition section.

4.1.1.   Attributes

The <layout> element has no attributes.

4.1.2.  Nested Elements

The <layout> element can contain one or more of the following nested elements.

Element Name

Description

Required

container

This element details a container into which a defined sub-set of parameters can be grouped for display.

Yes

4.2.    <container> Element

The <container> element is used to define a physical component into which a sub-set of parameters can be grouped for display purposes.

4.2.1.  Attributes

The <container> element has the following attributes:

 

Attribute Name

Description

Required

name

String detailing a user-friendly name for the container. This may be used by a MMI for display to a user.

Yes

type

String detailing the container type. Valid options are as follows:

TAB - All parameters referenced in this element will be positioned on the same Tab component in the MMI.

Yes

description

String detailing a user-friendly description for the container. This may be used by a MMI for display to a user.

No

4.2.2. Nested Elements

The <container> element can contain any amount of the following nested elements, and in any order. As a minimum, at least one element must be supplied.

 

The order in which the elements are defined here is the order in which the relevant parameters will appear within the container.

Element Name

Description

Required

fieldLayout

This element details the layout of a standard data or choice parameter.

No

structureLayout

This element details the layout of a structure parameter.

No

4.3.    <fieldLayout> Element

The <fieldLayout> element is used to define the layout of a standard data parameter (see Section 3.2), or a choice parameter (see Section 3.3).

4.3.1.  Attributes

The <fieldLayout> element has the following attributes:

 

Attribute Name

Description

Required

fieldId

String detailing the parameter id to be displayed.

 

The parameter id must exactly match one that has been specified in the field definition section.

Yes

type

String detailing the MMI component to be used in visualising the parameter. Valid options are as follows:

TEXT - A basic text field component.
DROPDOWN - A drop-down menu (sometimes known as a combo-box).

Note that a DROPDOWN component can only be specified for choice parameters of type SINGLE.

Yes

4.3.2. Nested Elements

The <fieldLayout> element has no nested elements.

4.4.   <structureLayout> Element

The <structureLayout> element is used to define the layout of a structure parameter only (see Section 3.5).

 

This layout is currently always shown as a table component in the MMI, irrespective of how many rows the structure is set to have.

4.4.1. Attributes

The <structureLayout> element has the following attributes:

 

Attribute Name

Description

Required

fieldId

String detailing the parameter id to be displayed.

 

The parameter id must exactly match one that has been specified in the field definition section.

Yes

4.4.2.     Nested Elements

The <structureLayout> element must contain exactly one <fieldLayout> element for every sub-parameter of the structure as defined in the field definition section (see Section 3.5.2).

 

The order in which the elements are defined here is the order in which the relevant sub-parameters will appear within the table component (and may therefore differ from the order defined in the field definition section).

 

Element Name

Description

Required

fieldLayout

This element details the layout of a standard data or choice parameter, which itself must be a recognised sub-parameter of the structure.

 

This is as defined in Section 4.3.

Yes

 

5.         Output Definition Section Up

The output definition section details how the template parameters are to be written to files for sending to a terminal.

 

The following rules are applicable to the output section:

 

                     The generated output can consist of one or more files

                     A parameter may be written in as many different locations as required (either in the same file, as well as across multiple files).

                     All parameters written (or defined free text) are done so on a separate line in the file.

 

The output definition section is usually created by the terminal application developer.

5.1.      <output> Element

This is the main element for the layout definition section.

5.1.1.   Attributes

The <output> element has no attributes

5.1.2.  Nested Elements

The <output> element has zero or more of the following nested elements

 

Element Name

Description

Required

propertyParamFile

This element details the generation of one parameter file.

Yes

5.2.     <propertyParamFile> Element

The <propertyParamFile> element is used to define the layout of one distinct file to be generated for sending to a terminal.

5.2.1.  Attributes

The <propertyParamFile> element has the following attributes:

 

Attribute Name

Description

Required

destination

String detailing the final destination of the file in the terminal. The actual format is specific to the terminal type being targeted by the template (a current restriction is that a template can only be targeted to one terminal type).

 

Telium Terminal Format: "<volume>:<filename>", or "filename" (with an assumed volume of SWAP).

Yes

propertyValueSep

String to be placed between the property name and the property value.

 

The default value is "=".

No

valueEnclosingString

String to be placed immediately before and after the property value.

 

e.g."&quot;" will give:

myParamId="myParamValue"

    

The default value is nothing.

No

beginLineString

String to be placed at the beginning of each line on which a parameter is written.

 

The default value is nothing.

No

endLineString

String to be placed at the end of each line on which a parameter is written.

 

The default value is nothing.

No

5.2.2. Nested Elements

The <propertyParamFile> element can contain any amount of the following nested elements, and in any order. As a minimum, at least one element must be supplied.

 

The order in which the elements are defined here is the order in which the relevant parameters and text will be written to the file.

Element Name

Description

Required

fieldOutput

This element details the writing of a standard data parameter to the file.

No

structureOutput

This element details the writing of a structure parameter to the file.

No

freeOutput

This element details the writing of custom 'free text' to the file.

No

5.3.     <fieldOutput> Element

The <fieldOutput> element is used to define the writing of a standard data parameter (see Section 3.2), or a choice parameter (see Section 3.3) to the file.

5.3.1.  Attributes

The <fieldOutput> element has the following attributes:

 

Attribute Name

Description

Required

fieldId

String detailing the parameter id to be output. This is also used as the property name that will be written in the file unless the propertyValue attribute is set.

The parameter id must exactly match one that has been specified in the field definition section.

Yes

propertyValue

String detailing the value to be written as the property name in the file .

 

Unless specified, the fieldId will be written as the property name.

No

writeEmptyValue

Boolean which indicates whether or not the parameter should be written to the file if it has an empty value ("").

 

The default value is false.

No

writeNullValue

Boolean which indicates whether or not the parameter should be written to the file if it has a null value.

 

The default value is false.

No

5.3.2. Nested Elements

The <fieldOutput> element has no nested elements.

5.4.    <structureOutput> Element

The <structureOutput> element is used to define the layout of a structure parameter only (see Section 3.5).

 

All values for the structure's sub-parameters will be written in the order as defined in the field definition section (see Section 3.5.2).

5.4.1.  Attributes

The <structureOutput> element has the following attributes:

 

Attribute Name

Description

Required

fieldId

String detailing the parameter id to be output. This is also used as the property name that will be written in the file unless the propertyValue attribute is set.

The parameter id must exactly match one that has been specified in the field definition section.

Yes

propertyValue

String detailing the value to be written as the property name in the file .

 

Unless specified, the fieldId will be written as the property name.

No

writeEmptyValue

Boolean which indicates whether or not sub-parameter values should be written to the file if it has an empty value ("").

 

The default value is false.

No

writeNullValue

Boolean which indicates whether or not sub-parameter values should be written to the file if it has a null value.

 

The default value is false.

No

fieldSeparator

String detailing the separator between sub-parameter values written.

 

e.g. "." will give:

myStructureId=subParameter1Value.subParameter2Value.subParameter3Value

 

Care should be taken to ensure that the sub-parameter value cannot contain this string itself.

 

The default value is "." (dot).

No

indexSeparator

String detailing the separator between the property name written, and the structure row index.

 

e.g. "." will give:

myStructureId.1=<values...>

myStructureId.2=<values...>

 

The default value is "." (dot).

No

indexFixedSize

Integer detailing the number of digits used to represent the index. If an index number does not use this amount of digits, then it will be left-padded with 0's.

 

Note that the index is only written for structures with more than one row.

 

The value -1 means that no index has to be written.

The value 0 means that natural numbers are written (with no formatting).

Any other value means to format the index to the given size. If the index number exceeds the given size then no formatting is done and the index number is written out as is.

 

e.g. "3" will give:

myStructureId.001=<values...>

myStructureId.002=<values...>

 

In all cases, the Index starts with value 1.

No

flatOutput

Boolean detailing whether or not structures have to be written with a sub-parameter value per line (instead of all sub-parameter values for a row appearing on the same line).

 

e.g.

"false" will give:

myStructureId.1=subParameter1Value.subParameter2Value.subParameter3value

myStructureId.2=subParameter1Value.subParameter2Value.subParameter3value

 

"true" will give:

myStructureId.subParameter1Id.1=subParameter1Value

myStructureId.subParameter2Id.1= subParamete2Value

myStructureId.subParameter3Id.1= subParameter3Value myStructureId.subParameter1Id.2=subParameter1Value

myStructureId.subParameter2Id.2= subParamete2Value

myStructureId.subParameter3Id.2= subParameter3Value

 

The default value is false.

 

No

5.4.2. Nested Elements

The <structureOutput> element has no nested elements.

5.5.     <freeOutput> Element

The <freeOutput> element is used to define a custom string of text to be written to the file.

5.5.1.  Attributes

The <freeOutput> element has the following attributes:

 

Attribute Name

Description

Required

output

String detailing the custom text to be written to the file.

Yes

5.5.2. Nested Elements

The <freeOutput> element has no nested elements.

6.         Small Parameter Template XML Example Up

This section gives a small (and contrived) example of how a parameter template XML file can be constructed. It also shows screenshots demonstrating how the layout will be rendered by the IngEstate Parameter Management MMI, and also how the relevant parameter files will be generated for the terminal.

6.1.     The Template XML

The example parameter template XML file is as follows. It defines a variety of different field types, multiple tabs for the MMI, and multiple as the output for the terminal.

 

<?xml version="1.0" encoding="utf-8"?>

 

<template id="example" name="Example Template" description="Example Template For IngEstate Parameter management" version="12345" xmlns="http://www.ingenico.com/ingestate/paramTemplate">

 

<fields>

<dataField id="account" name="Account Number" description="Full customer account number" minLength="5" maxLength="5" type="numeric" access="READWRITE"/>

<dataField id="internal" name="Ref ID" description="Customer internal reference id" defaultValue="To Be Assigned" access="READONLY"/>

<choiceField id="title" name="Title" description="Customer title" type="SINGLE" access="READWRITE">

<choiceValue label="Mr" value="0"/>

<choiceValue label="Mrs" value="1"/>

<choiceValue label="Ms" value="2"/>

<choiceValue label="Master" value="3"/>

</choiceField>

<structureField id="name" name="Name" description="Customer name" access="READWRITE" minRows="1" maxRows="1">

<dataField id="first" name="First Name" description="Customer first name" type="letters" access="READWRITE"/>

<dataField id="middle" name="Middle Name(s)" description="Customer middle name(s)" type="letters" access="READWRITE"/>

<dataField id="last" name="Surname" description="Customer surname" type="letters" access="READWRITE"/>

</structureField>

<dataField id="dob" name="Date Of Birth" description="Customer date of birth (dd-mm-yyyy)" dateTimeFormat="dd-MM-yyyy" access="READWRITE"/>

<structureField id="phone" name="Phone" description="Customer phone details" access="READWRITE" mandatory="false" maxRows="10">

<choiceField id="type" name="Type" description="Phont type" type="SINGLE" access="READWRITE">

<choiceValue label="Mobile" value="M"/>

<choiceValue label="Home" value="H"/>

<choiceValue label="Work" value="W"/>

</choiceField>

<dataField id="number" name="Number" description="Full number (including area code)" type="numeric" access="READWRITE"/>

</structureField>

 

<dataField id="8bits" name="8 Bits" description="Example of how 8 bits may be inserted and validated" type="alphanumeric" minLength="8" maxLength="8" regularExp="[01]{8}" access="READWRITE"/>

<dataField id="hex" name="Hex Value" description="2 digit hex value" type="hexadecimal" minLength="2" maxLength="2" access="READWRITE" mandatory="true"/>

</fields>

 

<layout>

<container name="Customer" type="TAB" description="Personal information regarding the customer">

<fieldLayout fieldId="account" type="TEXT"/>

<fieldLayout fieldId="internal" type="TEXT"/>

<fieldLayout fieldId="title" type="DROPDOWN"/>

<structureLayout fieldId="name">

<fieldLayout fieldId="first" type="TEXT"/>

<fieldLayout fieldId="middle" type="TEXT"/>

<fieldLayout fieldId="last" type="TEXT"/>

</structureLayout>

<fieldLayout fieldId="dob" type="TEXT"/>

<structureLayout fieldId="phone">

<fieldLayout fieldId="type" type="DROPDOWN"/>

<fieldLayout fieldId="number" type="TEXT"/>

</structureLayout>

</container>

 

<container name="Other Examples" type="TAB" description="Some random fields showing different input and validation examples">

<fieldLayout fieldId="8bits" type="TEXT"/>

<fieldLayout fieldId="hex" type="TEXT"/>

</container>

</layout>

 

<output>

<propertyParamFile destination="customer.par" valueEnclosingString="&quot;">

<fieldOutput fieldId="account"/>

<fieldOutput fieldId="title"/>

<structureOutput fieldId="name"/>

<fieldOutput fieldId="dob"/>

</propertyParamFile>

 

<propertyParamFile destination="phone.par" valueEnclosingString="&quot;">

<structureOutput fieldId="name"/>

<structureOutput fieldId="phone"/>

</propertyParamFile>

</output>
